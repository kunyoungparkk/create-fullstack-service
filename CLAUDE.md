# 코드 작성 원칙

LLM 코딩에서 흔한 실수를 줄이기 위한 행동 지침이다. 프로젝트 고유 원칙과 함께 적용한다.

> 트레이드오프: 이 지침은 속도보다 신중함에 무게를 둔다. 사소한 작업은 판단에 맡긴다.

## 1. 코딩 전에 생각한다

가정하지 말고, 혼란을 숨기지 말고, 트레이드오프를 드러낸다.

- 가정은 명시적으로 말한다. 불확실하면 묻는다.
- 해석이 여럿이면 모두 제시한다. 말없이 하나를 고르지 않는다.
- 더 단순한 방법이 있으면 말한다. 필요하면 반론한다.
- 불분명하면 멈춘다. 무엇이 혼란스러운지 짚고 묻는다.

## 2. 단순함을 우선한다

문제를 푸는 최소한의 코드만 쓰고, 추측성 코드는 두지 않는다.

- 요청하지 않은 기능은 만들지 않는다.
- 한 번만 쓰는 코드에 추상화를 두지 않는다.
- 요청하지 않은 유연성이나 설정 가능성을 넣지 않는다.
- 일어날 수 없는 상황에 대한 에러 처리를 넣지 않는다.
- 200줄을 썼는데 50줄로 가능하면 다시 쓴다.

스스로 묻는다. "시니어 엔지니어가 이걸 과하다고 할까?" 그렇다면 단순화한다.

## 3. 외과적으로 변경한다

꼭 필요한 것만 건드리고, 내가 만든 것만 치운다.

- 주변 코드·주석·포매팅을 "개선"하지 않는다.
- 망가지지 않은 것을 리팩터링하지 않는다.
- 내 취향과 달라도 기존 스타일을 따른다.
- 관련 없는 죽은 코드는 삭제하지 말고 언급한다.
- 내 변경으로 안 쓰이게 된 임포트·변수·함수만 제거한다.

기준: 바뀐 모든 줄이 사용자의 요청으로 곧장 이어져야 한다.

## 4. 목표 기반으로 실행한다

성공 기준을 정의하고, 검증될 때까지 반복한다.

- "검증 추가" → "잘못된 입력에 대한 테스트를 쓰고 통과시킨다"
- "버그 수정" → "버그를 재현하는 테스트를 쓰고 통과시킨다"
- "X 리팩터링" → "전후로 테스트가 통과하는지 확인한다"

강한 성공 기준은 독립적으로 반복하게 해준다. 약한 기준("일단 되게")은 끊임없는 확인을 부른다.

## 안티패턴 요약

| 원칙         | 안티패턴                             | 교정                                    |
| ------------ | ------------------------------------ | --------------------------------------- |
| 코딩 전 사고 | 형식·필드·범위를 말없이 가정         | 가정을 명시하고 확인 요청               |
| 단순함 우선  | 일회성 코드에 전략 패턴              | 복잡성이 실제로 필요할 때까지 함수 하나 |
| 외과적 변경  | 버그 고치며 따옴표·타입힌트까지 변경 | 보고된 문제를 고치는 줄만 변경          |
| 목표 기반    | "코드를 검토하고 개선하겠다"         | "테스트 작성 → 통과 → 회귀 없음 확인"   |

> 핵심: 좋은 코드는 내일의 문제를 미리 푸는 코드가 아니라, 오늘의 문제를 단순하게 푸는 코드다.

## 프로젝트 고유 원칙

karpathy 4원칙에 더해 이 저장소에서 지킨다.

- 검증된 패키지를 활용하고, 패키지의 기본값과 동일한 옵션은 명시하지 않는다.
- 코드 작성은 글쓰기와 같다. 문맥에 맞게 줄을 띄워 논리 흐름과 시각적 편안함을 살린다.

# 저장소 구조

Monorepo를 pnpm의 Workspace와 Turborepo를 사용하여 구성한다. 기술 스택, 명령어, 컨벤션은 각 Workspace의 `CLAUDE.md`를 참고한다.

| Workspace | 경로             |
| --------- | ---------------- |
| Server    | `apps/server`    |
| Client    | `apps/client`    |
| Types     | `packages/types` |

`packages/types`는 클라이언트와 서버가 공유하는 요청·응답 타입(와이어 모양)을 둔다. 서버 DTO는 이를 `implements`하고 응답의 날짜는 ISO 문자열로 직렬화하며, 클라이언트는 `import type`으로 사용한다. 검증 로직은 각 레이어가 담당한다.

# 개발 환경

- 런타임 환경은 Mise로 관리하며 구축과 실행은 `README.md`를 참고한다.
- 런타임 버전은 `mise.toml`에 정의돼 있다.
- 패키지 매니저는 pnpm만 사용하며 npm은 차단된다.
- `mise infra up --env <development|test|production>` 은 `apps/server/.env.<env>` 의 환경 변수를 적용해 인프라를 기동한다.

# 패키지 추가

- AI 에이전트가 새 패키지를 추가할 때는 먼저 사용자에게 제안하고 승인을 받는다.
- 사용자가 작성한 코드를 AI 에이전트가 다른 패키지로 대체할 수 있으면 사용자에게 제안하고 승인을 받는다.
- AI 에이전트는 공신력 있거나 유명한 패키지를 추천한다.

# 이슈 기반 작업 흐름

모든 작업은 **착수 전 GitHub 이슈**가 있어야 한다. 이슈가 그 작업의 단일 출처(SSOT)다.
기능 명세(예: `TODO_01_01`, `LOGIN_01_03`)는 이 문서가 아니라 **이슈 본문**에 둔다 —
불명확하면 추측하지 말고 사용자에게 요청한다.

**진입 경로**

- **기존 이슈**: 사용자가 이슈 링크/번호를 제공하면 `gh issue view <번호>`로 본문을 읽고, 그 이슈를 명세로 삼아 구현한다. 추측 금지 — 불명확하면 사용자에게 질문한다
- **신규 작업**: 이슈가 없으면 **먼저 `superpowers:brainstorming`으로 요구사항을 정리**한다. 사용자가 설계를 승인하면, 그 결과를 알맞은 이슈 템플릿에 채워 Claude가 `gh issue create`로 등록한 뒤 그 이슈로 구현한다

**brainstorming 연계** — 승인된 설계는 별도 spec 문서가 아니라 **GitHub 이슈 본문**으로 남긴다. 설계 승인 → 이슈 생성 → `writing-plans` → `subagent-driven-development` →
`verification-before-completion` 순서

**구현 실행** — `subagent-driven-development`로 계획을 실행한다. 태스크마다 새 서브에이전트가 `test-driven-development`로 구현하고 리뷰를 거친다. 계획이 필요 없는 소규모 작업은 `test-driven-development`로 직접 구현한다.

**템플릿 매핑** (`.github/ISSUE_TEMPLATE/`)

- `feat` → `feature_request.md`
- `fix` → `bug_report.md`
- `chore` → `chore-request.md`

**연결**

- 브랜치·커밋·PR은 해당 이슈를 참조한다 (PR 템플릿 `## 관련 이슈`에 링크, 커밋/PR 본문에 `#<번호>`)
- 이슈 → `feature/*` 브랜치 → `dev` 대상 PR → 머지 시 이슈 종료
